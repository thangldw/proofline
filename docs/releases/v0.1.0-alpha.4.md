# Proofline v0.1.0-alpha.4

This experimental pre-alpha release adds safe, observable polling for explicitly registered local
folders and hardens deterministic ingestion against redundant and partial reads. It is not stable
or production-supported.

## Highlights

- Set `PROOFLINE_FOLDER_WATCH_INTERVAL_SECONDS` from 1 through 3600 to scan registered roots once
  at API startup and then at the configured interval; `0` keeps the watcher disabled by default.
- `GET /api/v1/folder-watch` reports process-local lifecycle timestamps, counts, and stable error
  codes without exposing root paths, source content, prompts, or exception messages.
- Watcher roots use fresh database sessions. Watcher cycles and manual folder scans share one
  process-local coordinator, so they execute sequentially instead of racing source identity and
  immutable version creation.
- Watcher scans always preview missing files with `delete_missing=false`. Deletion remains a
  separate exact-set confirmed operation.
- Files whose identity, size, or timestamps change between pre-read and post-read `stat` calls fail
  with `file_changed_during_read`; no source version, evidence, or ingestion job is changed.
- Unchanged files are matched by current immutable-version content hash before job creation, so
  repeated scans no longer add redundant ingestion jobs.

## Run the watcher

```bash
export PROOFLINE_IMPORT_ROOTS=/absolute/path/to/vault
export PROOFLINE_FOLDER_WATCH_INTERVAL_SECONDS=30
proofline serve
curl http://localhost:8000/api/v1/folder-watch
```

The shipped Compose definition does not mount a host vault. Container users must add an explicit
read-only bind mount and register the corresponding container path.

## Assets

- Python wheel and source distribution for the local API/CLI.
- Optimized, unhosted web archive requiring a same-origin reverse proxy for `/api` and `/health`.
- `SHA256SUMS` covering every uploaded build artifact.

## Important limitations

- Polling is process-local, not a native filesystem notification service or multi-worker
  coordinator. Enable it in at most one API process for a vault.
- Clean shutdown waits for an active scan and its database transaction to finish; a large, slow, or
  unavailable filesystem can delay termination.
- The API has no authentication or multi-user authorization and binds to loopback by default.
- Repository Security plugin review, real-model comparison, external human pilot metrics, Windows
  verification, reranking, scalable vector indexing, and production qualification remain open.
- Desktop packaging, cloud sync, and production support are not included.

See the root README and backup/recovery guide before using approved data.
